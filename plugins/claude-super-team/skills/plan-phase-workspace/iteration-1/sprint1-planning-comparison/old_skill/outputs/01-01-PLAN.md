  <action>
Set up Drizzle ORM with PostgreSQL (use drizzle-orm and postgres packages).

Create schema.ts with three tables:

**workspaces**: id (uuid, PK, default gen_random_uuid()), name (text, not null), slug (text, unique, not null), created_at (timestamp, default now()), updated_at (timestamp).

**users**: id (uuid, PK), email (text, unique, not null), name (text), password_hash (text, nullable -- needed for OAuth-only users), avatar_url (text), created_at, updated_at.

**memberships**: id (uuid, PK), user_id (uuid, FK -> users), workspace_id (uuid, FK -> workspaces), role (text, not null, check constraint for 'owner' | 'admin' | 'viewer'), invited_email (text, nullable -- for pending invites), accepted_at (timestamp, nullable), created_at. Add unique constraint on (user_id, workspace_id).

In index.ts, create the db connection using postgres() driver and drizzle(). Also export a `tenantDb` helper function that takes a workspace_id and returns a query wrapper that automatically filters by workspace_id. This is the primary isolation mechanism -- all workspace-scoped queries go through tenantDb.

Generate the migration SQL file from the schema. Set up drizzle.config.ts pointing to the PostgreSQL connection string from DATABASE_URL env var.
  </action>
  <verify>
Run `npx drizzle-kit generate` and confirm it produces the migration file. Run `npx drizzle-kit push` against a local PostgreSQL database and confirm all three tables are created. Query the information_schema to verify unique constraints exist on workspaces.slug and memberships(user_id, workspace_id).
  </verify>
  <done>All three tables (workspaces, users, memberships) exist in PostgreSQL with correct columns, types, constraints, and foreign keys. The tenantDb helper is exported and scopes queries by workspace_id.</done>
</task>

<task type="auto">
  <name>Task 2: Workspace resolution middleware and creation API</name>
  <files>
    src/middleware.ts

  <action>
Create Next.js middleware at src/middleware.ts that handles workspace resolution. The URL pattern is `/w/{slug}/...`. The middleware:
1. Extracts the workspace slug from the URL path
2. Looks up the workspace by slug in the database
3. If not found, returns 404
4. Injects the workspace_id into request headers (x-workspace-id) so downstream API routes and server components can read it
5. Matcher config should apply to `/w/:path*` routes and `/api/:path*` routes

Create POST /api/workspaces endpoint that:
1. Accepts { name, slug } in the request body
2. Validates slug format (lowercase alphanumeric + hyphens, 3-50 chars)
3. Creates the workspace row
4. Creates a membership row with role='owner' linking the creating user
5. Returns the created workspace with 201 status

Create a minimal onboarding page at src/app/(app)/onboarding/page.tsx with a form for workspace name and slug. On submit, calls POST /api/workspaces and redirects to /w/{slug}/dashboard. This is a server-rendered page with a client form component. Keep the UI functional but minimal -- styling is not the focus.
  </action>
  <verify>
Start the Next.js dev server. Navigate to /onboarding, fill in a workspace name and slug, and submit. Confirm the workspace is created in the database. Navigate to /w/{slug}/dashboard and confirm the middleware resolves the workspace (check x-workspace-id header). Navigate to /w/nonexistent and confirm 404. Attempt to create a workspace with a duplicate slug and confirm it returns a 409 conflict error.
  </verify>
  <done>Workspace creation flow works end-to-end: form submission creates workspace + owner membership in DB. Middleware resolves workspace slug from URL and injects workspace_id. Invalid slugs return 404, duplicate slugs return 409.</done>
</task>
