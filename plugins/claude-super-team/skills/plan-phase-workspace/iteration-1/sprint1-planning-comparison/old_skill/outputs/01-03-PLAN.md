  <name>Task 1: Role permission system and member management API</name>
  <files>
    src/lib/auth/roles.ts

  <action>
**roles.ts**: Define the role hierarchy and permission system.

Role hierarchy: owner > admin > viewer.

Permission matrix:
- `canManageMembers`: owner, admin (invite, change roles below own level, remove members below own level)
- `canModifySettings`: owner, admin
- `canManageBilling`: owner only
- `canViewData`: owner, admin, viewer

Export a `requireRole(...roles)` function that returns a middleware/guard. It reads the current user_id (from session) and workspace_id (from middleware headers), looks up the membership, and checks if the user's role is in the allowed list. Returns 403 if not authorized. Also export individual permission check functions.

Important constraint: An admin cannot change another admin's role or remove them. An admin cannot promote anyone to owner. Only owners can manage other admins. Implement this as a `canActOn(actorRole, targetRole)` check.

**GET /api/workspaces/[slug]/members**: Returns all memberships for the workspace (including pending invitations). Any authenticated workspace member can access this. Include user name, email, role, and accepted_at status.

**POST /api/workspaces/[slug]/members**: Invite a new member. Requires admin or owner role (use requireRole). Accepts { email, role }. Role must be 'admin' or 'viewer' (cannot invite as owner). Creates a membership row with invited_email set and user_id null, generates a unique invitation token (store in a new `invitation_token` column on memberships or a separate invitations table -- use a separate invitations table with: id, workspace_id, email, role, token (uuid), expires_at (7 days), accepted_at). Returns the invitation details. If the email is already a member, return 409.

**PATCH /api/workspaces/[slug]/members/[memberId]**: Change a member's role. Requires canActOn check -- the actor must have a higher role than both the target's current role and the new role. Accepts { role }. Cannot change the last owner's role (at least one owner must remain).

**DELETE /api/workspaces/[slug]/members/[memberId]**: Remove a member. Requires canActOn check. Cannot remove the last owner.
  </action>
  <verify>
Test the permission matrix with curl:
1. As owner: invite user with role viewer -- expect 201
2. As owner: invite user with role admin -- expect 201
3. As viewer: attempt to invite -- expect 403
4. As admin: invite user with role viewer -- expect 201
5. As admin: attempt to invite with role owner -- expect 403
6. As owner: change a viewer to admin -- expect 200
7. As admin: attempt to change another admin's role -- expect 403
8. As owner: remove a viewer -- expect 200
9. As admin: attempt to remove another admin -- expect 403
10. Attempt to remove the only owner -- expect 400
  </verify>
  <done>Role permission system enforces owner > admin > viewer hierarchy. Members can be invited with specific roles. Role changes and removal respect the hierarchy (cannot act on same-level or above). Last owner protection is in place.</done>

**POST /api/invitations/[token]/accept**: Public endpoint (but requires authentication). Looks up the invitation by token. If expired (past expires_at) or already accepted, return 400. If the authenticated user's email matches the invited_email, create the membership (set user_id, accepted_at on membership row or create membership from invitation record). Return 200 with workspace slug so the client can redirect. If emails don't match, return 403.

**Team management page** at /w/[slug]/settings/members:
- Server component that fetches the current user's role in this workspace
- If the user is a viewer, show the member list in read-only mode (no invite button, no role change controls, no remove buttons)
- If admin or owner, show:
  - Member list table: name, email, role, status (active/pending), with role dropdown and remove button per row (respecting canActOn rules -- disable controls for members at or above the current user's role)
  - Invite form: email input, role selector (only roles below the current user's level), invite button
  - Pending invitations section showing outstanding invites with option to revoke
- Use basic Tailwind styling. Client component for interactive parts (invite form, role changes, remove actions).
- Actions call the API routes defined in Task 1 and refresh the member list on success.
  </action>
