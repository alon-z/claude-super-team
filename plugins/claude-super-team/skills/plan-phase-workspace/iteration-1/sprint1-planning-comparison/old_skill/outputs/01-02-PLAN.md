Purpose: Authentication is required before any workspace-scoped feature can identify the acting user. This plan adds email/password and Google OAuth -- the two most common auth methods.
Output: Auth library (session + passwords), signup/login/logout API routes, Google OAuth flow, and login/signup UI pages.
</objective>

  <action>
Create the auth library:

**session.ts**: Use the `jose` library (NOT jsonwebtoken -- it has CommonJS issues with Next.js Edge runtime) for JWT handling. Implement:
- `createSession(userId: string)`: Creates a JWT with user_id claim, 7-day expiry, signs with JWT_SECRET env var. Sets it as an httpOnly, secure, sameSite='lax' cookie named 'session'.
- `getSession()`: Reads the session cookie, verifies the JWT, returns { userId } or null if invalid/expired.
- `clearSession()`: Removes the session cookie.

**passwords.ts**: Use bcrypt. `hashPassword(plain)` returns the hash. `verifyPassword(plain, hash)` returns boolean.

**POST /api/auth/signup**: Accepts { email, password, name }. Validates email format and password minimum 8 chars. Checks if email already exists (return 409 if so). Hashes password, inserts user into users table, creates session, returns 201 with user data (excluding password_hash).

**POST /api/auth/login**: Accepts { email, password }. Looks up user by email. If not found or password doesn't match, return 401 with generic "Invalid credentials" message (no email enumeration). Creates session, returns 200 with user data.

**POST /api/auth/logout**: Calls clearSession(), returns 200.

Update src/middleware.ts to also call getSession() for authenticated routes. If no valid session on /w/* or /api/* routes (except /api/auth/*), redirect to /login.
  </action>
  <verify>
Use curl to test the full flow:
1. `curl -X POST /api/auth/signup -d '{"email":"test@test.com","password":"password123","name":"Test"}' -c cookies.txt` -- expect 201 with Set-Cookie header
2. `curl -X POST /api/auth/login -d '{"email":"test@test.com","password":"password123"}' -c cookies.txt` -- expect 200 with Set-Cookie
3. `curl /api/workspaces -b cookies.txt` -- expect 200 (authenticated)
4. `curl /api/workspaces` -- expect redirect to /login (no cookie)
5. `curl -X POST /api/auth/login -d '{"email":"test@test.com","password":"wrong"}' ` -- expect 401
6. `curl -X POST /api/auth/logout -b cookies.txt -c cookies.txt` then `curl /api/workspaces -b cookies.txt` -- expect redirect to /login
  </verify>
  <done>Users can sign up with email/password (201 + session cookie), log in (200 + session cookie), and log out (session cleared). Invalid credentials return 401. Unauthenticated requests to protected routes redirect to /login. Sessions persist across requests via httpOnly JWT cookie.</done>

  <action>
**Google OAuth flow:**

GET /api/auth/google: Generate a random state parameter, store it in a short-lived httpOnly cookie. Redirect to Google's OAuth authorization URL with client_id (GOOGLE_CLIENT_ID env var), redirect_uri (/api/auth/google/callback), scope (openid email profile), and state parameter.

GET /api/auth/google/callback: Verify state parameter matches cookie. Exchange the authorization code for tokens using Google's token endpoint. Fetch user profile from Google's userinfo endpoint. Look up user by email -- if exists, create session and redirect to /w/ (their most recent workspace) or /onboarding. If not exists, create user record (no password_hash, name and avatar from Google profile), create session, redirect to /onboarding.

**Auth UI pages:**

Create an (auth) route group layout at src/app/(auth)/layout.tsx with centered card styling -- a simple centered container, nothing elaborate.

Login page: Email and password form fields, submit button that POSTs to /api/auth/login. On success redirect to /w/ or /onboarding. "Sign in with Google" button that navigates to /api/auth/google. Link to signup page at bottom.

Signup page: Name, email, and password form fields, submit button that POSTs to /api/auth/signup. On success redirect to /onboarding. "Sign up with Google" button. Link to login page at bottom.

Both pages should be client components for form handling. Use basic HTML form elements with minimal Tailwind styling. Show error messages from API responses inline.
