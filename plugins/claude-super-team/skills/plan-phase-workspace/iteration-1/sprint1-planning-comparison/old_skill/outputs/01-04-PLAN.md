  <action>
**GET /api/auth/saml/metadata**: Accepts ?workspace query param. Looks up SSO config for the workspace. Returns SP metadata XML with entity ID and ACS endpoint. Content-Type: application/xml. This endpoint is public so IdP admins can import it.

**GET /api/auth/saml/login**: Accepts ?workspace query param (workspace slug). Looks up SSO config. If not configured or not enabled, return 404. Calls createSAMLRequest to generate the redirect URL. Stores workspace_id in a short-lived cookie (saml_workspace) so the callback knows which workspace initiated the flow. Redirects to the IdP SSO URL.

**POST /api/auth/saml/callback**: Receives the SAML response from the IdP (form-encoded SAMLResponse parameter). Reads the saml_workspace cookie to identify the workspace. Calls parseSAMLResponse to validate and extract user profile. Then:
1. Look up user by email. If not found, create a new user record (no password_hash).
2. Check if user is already a member of the workspace. If not, create a membership with role 'viewer' (default role for SSO-provisioned users).
3. Create session using createSession() from the auth library.
4. Redirect to /w/{slug}/dashboard.

**SSO settings page** at /w/[slug]/settings/sso:
- Only accessible to workspace owners (check role, return 403 for others)
- Form with fields: IdP Entity ID, IdP SSO URL, IdP Certificate (textarea for PEM)
- Toggle to enable/disable SSO
- Display the SP Metadata URL that admins should import into their IdP
- On save, calls saveSSOConfig. Show success/error feedback.

**Update login page** (src/app/(auth)/login/page.tsx): Add a "Sign in with SSO" section below the Google button. Show a text input for "Workspace slug" and a button. When clicked, redirect to /api/auth/saml/login?workspace={slug}. This allows SSO users to initiate login from the main login page.
