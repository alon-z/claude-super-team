  <action>
Install drizzle-orm, drizzle-kit, and @neondatabase/serverless (or pg, depending on deployment -- use pg for Railway PostgreSQL).

Create the Drizzle schema file with three tables:

1. **workspaces**: id (uuid, PK), name (text, not null), slug (text, unique, not null), created_at (timestamp, default now). The slug is the workspace identifier used in URLs and tenant resolution.

2. **users**: id (uuid, PK), email (text, unique, not null), password_hash (text, nullable -- null for OAuth-only users), name (text), email_verified (boolean, default false), created_at (timestamp, default now).

3. **workspace_members**: id (uuid, PK), workspace_id (uuid, FK -> workspaces.id, not null), user_id (uuid, FK -> users.id, not null), role (text, not null, default 'viewer'), joined_at (timestamp, default now). Add a unique constraint on (workspace_id, user_id).

Create the Drizzle client in index.ts using the pg driver. Read the DATABASE_URL from environment variables. Export the db instance.

Generate the SQL migration using drizzle-kit. Ensure the migration creates all tables with proper foreign keys and indexes (index on workspace_members.workspace_id for tenant queries, index on users.email for login lookups).

Add a drizzle.config.ts at project root pointing to the schema file and migrations directory.
  </action>
  <verify>
Run `npx drizzle-kit generate` and confirm migration SQL is produced without errors. Run `npx drizzle-kit push` against a local PostgreSQL and verify all three tables are created with `psql -c "\dt"`. Confirm the unique constraint on workspace_members with `psql -c "\d workspace_members"`.
  </verify>
  <done>Three tables (workspaces, users, workspace_members) exist in the database with correct columns, foreign keys, indexes, and unique constraints. Drizzle client exports a working db instance.</done>

<task type="auto">
  <name>Task 2: Tenant resolution middleware and scoped queries</name>
  <files>
    src/lib/tenant.ts
    src/middleware.ts
  </files>
  <action>
Create tenant.ts with two exports:

1. **resolveWorkspace(request)**: Extracts workspace context from the request. Strategy: read workspace slug from a custom header (x-workspace-id) or from a cookie set at login. Returns the workspace record or null. This will be called by middleware and by API route handlers.

2. **withTenant(workspaceId)**: Returns a helper object that wraps Drizzle queries with an automatic `where workspace_id = workspaceId` filter. At minimum, provide a `query` method that takes a Drizzle table reference and returns a query builder pre-filtered by workspace_id. This ensures all data access is scoped by default -- callers never construct raw queries without tenant filtering.

Create Next.js middleware.ts at the project root. The middleware should:
- Skip tenant resolution for public paths: /login, /signup, /api/auth/*, /favicon.ico, /_next/*
- For all other routes, call resolveWorkspace(). If no workspace is resolved AND the user is not on a public path, redirect to /login.
- Store resolved workspace ID in a request header (x-workspace-id) so downstream API routes and server components can access it without re-resolving.

Use Next.js middleware `matcher` config to limit execution to relevant paths.
  </action>
  <verify>
Start the Next.js dev server. Verify that accessing /dashboard without auth redirects to /login. Verify that /login and /signup are accessible without workspace context. Add a temporary API route that reads x-workspace-id from headers and returns it -- confirm it reflects the resolved workspace.
  </verify>
  <done>Middleware resolves workspace context on every non-public request. Unauthenticated/no-workspace requests redirect to /login. withTenant helper enforces workspace-scoped queries by default. Public paths (/login, /signup, /api/auth/*) bypass tenant resolution.</done>
</task>
