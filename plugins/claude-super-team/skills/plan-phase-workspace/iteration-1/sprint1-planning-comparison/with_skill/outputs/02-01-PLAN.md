Create the standalone ingestion service with its own HTTP server and TimescaleDB schema for time-series metric storage.

Purpose: The ingestion service is a separate backend (not part of the Next.js app) that handles data source connections, metric ingestion, and scheduling. This plan establishes the service and its database foundation.
Output: A runnable Node.js HTTP service with TimescaleDB tables for metrics, data sources, and sync jobs.
</objective>

  <action>
Create a new directory at services/ingestion/ as a standalone Node.js project separate from the Next.js app.

Initialize package.json with dependencies: hono (lightweight HTTP framework -- preferred over Express for its small footprint and TypeScript-first design), drizzle-orm, pg, dotenv. Add dev dependencies: typescript, tsx, drizzle-kit, @types/pg.

Create tsconfig.json targeting ES2022 with NodeNext module resolution.

Create the main entry point using Hono. Set up:
- GET /health: Return { status: "ok", database: "connected" } after pinging the database. Return 503 if DB is unreachable.

The server should listen on PORT from env (default 3001) to avoid conflict with Next.js on 3000.

Create a Dockerfile for Railway deployment: Node 20 alpine, npm install, build TypeScript, run the compiled output.

Add npm scripts: "dev" (tsx watch src/index.ts), "build" (tsc), "start" (node dist/index.js).

The Hono server export pattern for this setup: `serve({ fetch: app.fetch, port })` using @hono/node-server.

Confirm `npm run build` compiles without TypeScript errors.

<task type="auto">
  <name>Task 2: TimescaleDB schema and migrations</name>
  <files>

    services/ingestion/src/db/migrations/001_timescale.sql

  <action>
Create the Drizzle client in db/index.ts, reading INGESTION_DATABASE_URL from env (separate database from the Next.js app's PostgreSQL, or the same PostgreSQL with TimescaleDB extension enabled).

Create the schema with three tables:

1. **data_sources**: id (uuid, PK), workspace_id (uuid, not null -- references the workspace in the main app DB conceptually, but no FK since separate databases), type (text, not null -- e.g., "stripe"), config (jsonb, not null -- encrypted credentials and connection details), refresh_interval (text, not null, default "5min" -- one of "1min", "5min", "15min", "manual"), status (text, not null, default "active"), created_at (timestamp, default now), updated_at (timestamp, default now).

2. **metrics**: time (timestamptz, not null), workspace_id (uuid, not null), source_id (uuid, FK -> data_sources.id, not null), metric_name (text, not null), metric_value (numeric, not null), dimensions (jsonb, default '{}' -- for key-value label pairs like { currency: "usd" }). This table becomes a TimescaleDB hypertable partitioned on the `time` column.

3. **sync_jobs**: id (uuid, PK), source_id (uuid, FK -> data_sources.id, not null), status (text, not null -- "pending", "running", "completed", "failed"), started_at (timestamptz), completed_at (timestamptz), records_synced (integer, default 0), error_message (text), created_at (timestamp, default now).

In the migration SQL, after creating the metrics table, add:
```sql
SELECT create_hypertable('metrics', 'time');
```
This converts it to a TimescaleDB hypertable. Also create an index on (workspace_id, source_id, metric_name, time DESC) for efficient dashboard queries.

Add a drizzle.config.ts in services/ingestion/ pointing to the local schema and migrations.
  </action>
  <verify>
Run `cd services/ingestion && npx drizzle-kit generate` and confirm migration SQL is produced. Run the migration against a local PostgreSQL with TimescaleDB extension: `psql -d ingestion -f src/db/migrations/001_timescale.sql`. Verify with `psql -c "\d metrics"` that the table exists and `psql -c "SELECT * FROM timescaledb_information.hypertables"` shows the metrics hypertable. Confirm data_sources and sync_jobs tables are created.
  </verify>
  <done>TimescaleDB schema has data_sources, metrics (hypertable), and sync_jobs tables. Metrics table is partitioned by time for efficient time-series queries. Drizzle client connects to the ingestion database independently.</done>
</task>
