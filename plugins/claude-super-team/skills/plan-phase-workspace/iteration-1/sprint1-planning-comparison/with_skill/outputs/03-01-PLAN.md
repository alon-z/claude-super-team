**Widget types (types.ts):**
Define shared types:
- MetricPoint: { time: string (ISO 8601), value: number }
- TimeSeriesData: { metricName: string, points: MetricPoint[], unit?: string }
- WidgetProps (base): { title: string, sourceId: string, metricName: string, refreshInterval?: number }
- LineChartProps extends WidgetProps
- BarChartProps extends WidgetProps with { bucketBy?: "day" | "week" | "month" }
- KpiCardProps extends WidgetProps with { format?: "currency" | "number" | "percent", trendPeriod?: "day" | "week" }

**Metrics client (metrics-client.ts):**
Create a client that communicates with the ingestion service to fetch metric data. The Next.js app calls the ingestion service via its internal API.

- fetchMetrics(sourceId, metricName, timeRange): Calls the ingestion service at INGESTION_SERVICE_URL/api/metrics?sourceId=X&metricName=Y&from=Z&to=W. Returns TimeSeriesData. This route doesn't exist yet on the ingestion side -- also create it.

Actually, since the ingestion service doesn't have a metrics query endpoint yet, create a Next.js API route as a proxy:
- Add a query endpoint to the ingestion service plan context note: the executor should add GET /api/metrics to the ingestion service (services/ingestion/src/routes/metrics.ts) that queries the TimescaleDB metrics hypertable with time_bucket() for aggregation, filtered by workspace_id, source_id, and metric_name. For now, include this in the metrics-client as a TODO comment and have the client call a Next.js API route instead.

Create a Next.js API route at src/app/api/metrics/route.ts that proxies to the ingestion service. It should:
- Read the workspace ID from the authenticated session
- Forward the request to INGESTION_SERVICE_URL/api/metrics with the workspace_id, sourceId, metricName, from, to params
- Pass the INGESTION_API_KEY as Authorization header
- Return the response to the client

The metrics-client.ts should call /api/metrics (the local Next.js proxy) from the browser. Export fetchMetrics() and fetchLatestMetric() (fetchLatestMetric just gets the most recent point).

Install recharts (React charting library -- well-maintained, good Next.js compatibility, renders to SVG).

**Line chart widget (line-chart.tsx):**
Client component ("use client"). Accepts LineChartProps. On mount and at refreshInterval, calls fetchMetrics() with the configured sourceId and metricName. Renders a Recharts LineChart with:
- X-axis: time labels (formatted as date/time)
- Y-axis: metric values with auto-scaling
- Tooltip showing exact value and timestamp on hover
- Responsive container that fills parent width
- Title displayed above the chart
- Loading skeleton while data is fetching
- Error state if fetch fails (show message, not crash)

**Bar chart widget (bar-chart.tsx):**
Client component. Accepts BarChartProps. Same data fetching pattern as line chart. Renders a Recharts BarChart with:
- X-axis: time buckets (bucketed by day/week/month based on bucketBy prop)
- Y-axis: aggregated metric values
- Tooltip on hover
- Responsive container
- Title, loading skeleton, error state

For time bucketing on the client side: group MetricPoints by the bucketBy period and sum values within each bucket. The ingestion service could do this server-side with TimescaleDB's time_bucket(), but client-side bucketing works fine for the initial viewer.

**KPI card widget (kpi-card.tsx):**
Client component. Accepts KpiCardProps. Calls fetchLatestMetric() to get the current value. Also fetches the value from one period ago (based on trendPeriod) to calculate trend. Renders:
- Large metric value, formatted based on `format` prop (currency: "$1,234.56", number: "1,234", percent: "12.3%")
- Title/label above the value
- Trend indicator: green up arrow if value increased vs prior period, red down arrow if decreased, gray dash if unchanged
- Trend percentage change displayed next to the arrow
- Loading skeleton, error state

All three widgets should handle empty data gracefully (show "No data available" message, not crash or show blank chart).
