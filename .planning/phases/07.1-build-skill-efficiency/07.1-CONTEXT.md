# Context for Phase 7.1: Build Skill Efficiency

## Phase Boundary (from ROADMAP.md)

**Goal:** Make /build significantly faster for well-understood projects by fixing teams detection, adding smarter pipeline depth, skipping verification when validation passes clean, and improving wave parallelism in plans
**Type:** Quick phase

**Success Criteria:**
1. Execute-phase reads teams-available from gather script PREFERENCES section instead of checking inaccessible env var (bug fix)
2. /build's adaptive pipeline depth heuristic accounts for project complexity class and tech stack coverage, not just keyword matching
3. Verification is skippable via a `verification` preference in STATE.md (always|on-failure|disabled), defaulting to on-failure
4. Plan-phase planner instructions produce more aggressive wave batching

**What's in scope:**
- Bug fix: execute-phase teams detection
- Build skill: improved adaptive pipeline depth heuristic in pipeline-guide.md + SKILL.md
- Verification preference: STATE.md, execute-phase gather-data.sh, execute-phase SKILL.md, build SKILL.md
- Planner wave batching: plan-phase planner instructions

**What's out of scope:**
- Cross-phase parallel DAG execution (captured in IDEAS.md for future)
- Execute-phase teams coordination / SendMessage between agents
- Telemetry or metrics changes

---

## Motivation

ez-pz-bank (6-phase CRUD app) built via /build took 4.5 hours. A 40-minute raw Claude session could produce nearly the same result. Root causes:

1. **Every phase ran FULL pipeline** (discuss + research + plan + execute) even for well-understood CRUD domains with a fully-specified tech stack in build-preferences.md. The heuristic's keyword matching triggered on "auth", "API", "dashboard" etc.
2. **Verification ran after every phase** producing 100-150 line reports that found only 1 meaningful bug total (env.ts not imported). Build+tests passing was sufficient signal.
3. **Plans had overly sequential waves** -- tasks that could safely run in parallel were serialized.

## Implementation Decisions

### Teams Detection Bug Fix

**Decision:** Change execute-phase SKILL.md Phase 2 (lines 91-96) to read `teams-available` from the pre-loaded PREFERENCES section output of gather-data.sh, instead of telling the LLM to check `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS` env var directly (which it cannot access).

**Root cause:** The gather script correctly outputs `teams-available: true` or `teams-available: false` on line 21, but the SKILL.md instructions tell the LLM to check the environment variable, which it has no access to. The `--team` flag works because it's parsed from $ARGUMENTS text.

**Fix:** Replace "Else if `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1` in environment" with "Else if `teams-available: true` in the pre-loaded PREFERENCES section".

### Verification Preference

**Decision:** Add `verification` preference to STATE.md alongside existing `execution-model` and `simplifier` preferences.

**Values:** `always` | `on-failure` | `disabled`
- `always`: current behavior -- run deep verification (VERIFICATION.md) after every phase execution
- `on-failure`: skip deep verification when build+tests pass clean; run verification only when build or tests fail (DEFAULT)
- `disabled`: never run deep verification

**How it flows:**
- User sets `verification: on-failure` (or always/disabled) in STATE.md directly
- /build reads it during preference resolution (same pattern as execution-model)
- Execute-phase gather-data.sh adds `grep -E '^verification:' .planning/STATE.md` to PREFERENCES section
- Execute-phase SKILL.md reads the preference and decides whether to spawn the verifier agent
- When `on-failure`: skip verifier if /build's build+tests passed; run verifier if they failed or were skipped

**Rationale:** STATE.md is the single source for project preferences. No need for build-preferences.md layer -- this is a per-project operational choice.

### Adaptive Pipeline Depth Improvements

**Decision:** Enhance the heuristic in `references/pipeline-guide.md` Section 2 to consider:

1. **Tech stack coverage**: If build-preferences.md specifies the full tech stack (framework, ORM, auth, UI) AND the phase uses only those technologies, bias toward SIMPLE. The current heuristic checks if "build-preferences.md does NOT specify tech decisions" but doesn't weight how much of the phase's domain is covered.

2. **Project complexity class**: Add a lightweight complexity assessment early in /build (after /create-roadmap) based on:
   - Number of phases (fewer = simpler)
   - Success criteria density (average criteria per phase)
   - External integration count (APIs, services, databases beyond the primary)
   - Whether the project is greenfield with a specified stack

   Classify as: `standard` (bias SIMPLE) or `complex` (bias FULL). Log in BUILD-STATE.md.

3. **Cumulative knowledge discount**: After phases 1-2 establish patterns (auth, DB, API conventions), later phases that follow the same patterns should lean SIMPLE even if they mention "new" domains that are actually just new routes/pages using established patterns.

**Constraints:** The default "if mixed, run FULL" stays -- but the signals should be better calibrated so fewer phases trigger mixed.

### Planner Wave Batching

**Decision:** Update plan-phase planner instructions to be more aggressive about parallel wave assignment.

**Current problem:** Planners tend to serialize plans into sequential waves even when they operate on independent file sets. For example, "create API routes" and "create UI components" often end up in wave 1 and wave 2 when they could both be wave 1.

**Fix:** Add explicit guidance to the planner reference/template:
- Plans that touch non-overlapping file sets should be same wave
- Plans that only share read-only dependencies (e.g., both read schema but don't modify it) should be same wave
- Only use sequential waves when a plan's OUTPUT is another plan's INPUT (e.g., plan A creates the DB schema, plan B writes migrations from that schema)
- When in doubt, same wave -- execute-phase handles coordination
