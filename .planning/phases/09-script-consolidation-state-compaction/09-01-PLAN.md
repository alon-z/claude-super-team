  <action>
Create `plugins/claude-super-team/scripts/gather-common.sh` as a sourceable library (not standalone). It must provide these functions:

**`emit_project_section`**: Emit `=== PROJECT ===` section. Accept an optional `$P` variable (defaults to `.planning`). Check `SKIP_PROJECT` env var -- if "1", print `(in context)`. Otherwise `cat $P/PROJECT.md` or print `(missing)`. Must also check file existence and emit `HAS_PROJECT=true/false`.

**`emit_roadmap_section`**: Same pattern for ROADMAP.md with `SKIP_ROADMAP`.

**`emit_state_section`**: Same pattern for STATE.md with `SKIP_STATE`. Must also check and emit `HAS_STATE=true/false`.

**`emit_phase_completion`**: Emit `=== PHASE_COMPLETION ===` section. Loop through `.planning/phases/*/`, count `*-PLAN.md` and `*-SUMMARY.md` files per directory, and emit status lines in the format: `{dir_name}|{status}|plans={N}|summaries={N}` where status is `complete` (summaries >= plans > 0), `partial` (summaries > 0 but < plans), `planned` (plans > 0, summaries == 0), or `empty`.

This is the exact logic currently duplicated in build/gather-data.sh (lines 68-84) and execute-phase/gather-data.sh (lines 46-62). Use the same output format.

**`emit_sync_check`**: Emit `=== SYNC_CHECK ===` section. Extract phase numbers from directories (strip leading zeros), phase numbers from ROADMAP.md, current phase from STATE.md, and CHECKED/UNCHECKED phase lists from ROADMAP.md checkboxes. This logic currently appears in execute-phase/gather-data.sh (lines 64-71) and progress/gather-data.sh (lines 92-111).

**`emit_preferences`**: Emit `=== PREFERENCES ===` section. Extract `execution-model`, `simplifier`, and `verification` values from STATE.md, with fallback defaults (`unset`). Check `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS` env var for teams-available. This logic currently appears in execute-phase/gather-data.sh (lines 19-22).

**`emit_structure`**: Emit `=== STRUCTURE ===` section. Check existence of PROJECT.md, ROADMAP.md, STATE.md, SECURITY-AUDIT.md and emit `HAS_*=true/false` for each. Also check for `.planning/codebase/` directory.

The script header should include:
```bash
#!/usr/bin/env bash
# gather-common.sh - Shared functions for skill gather-data.sh scripts
#
# Sourced by per-skill gather-data.sh via:
#   source "${CLAUDE_PLUGIN_ROOT}/scripts/gather-common.sh"
#
# All functions assume the working directory is the project root.
```

All functions must be safe to call independently. Each function emits its own `=== SECTION ===` header. Use `P="${P:-.planning}"` at the top so callers can override the planning directory path.

<task type="auto">
  <name>Task 2: Add create_phase_dir() to phase-utils.sh</name>
  <files>plugins/claude-super-team/scripts/phase-utils.sh</files>
  <action>
Add a `create_phase_dir()` function to the existing `plugins/claude-super-team/scripts/phase-utils.sh`. Place it after the existing `find_phase_dir()` function.

The function must:
1. Accept a raw phase number as the single argument (e.g., `9`, `2.1`, `02`)
2. Call `normalize_phase` on it to get the zero-padded form
3. First check if the directory already exists via `find_phase_dir` -- if so, print the existing path and return 0
4. If no directory exists, extract the phase name from ROADMAP.md by grepping for `Phase {raw_number}:` (handling both padded and unpadded), extracting the text after the colon, slugifying it (lowercase, spaces to hyphens, strip special chars, truncate to 40 chars)
5. Create the directory: `mkdir -p ".planning/phases/${padded}-${slug}"`
6. Print the created path to stdout
7. Return 0 on success, 1 if the phase is not found in ROADMAP.md

This replaces the inline pipeline currently in discuss-phase, plan-phase, quick-plan, and execute-phase SKILL.md files (approximately):
```bash
PHASE_NAME=$(grep "Phase ${PHASE_NUM}:" .planning/ROADMAP.md | sed 's/.*Phase [0-9]*: //' | sed 's/ *-.*//' | tr '[:upper:]' '[:lower:]' | tr ' ' '-')
mkdir -p ".planning/phases/${PHASE}-${PHASE_NAME}"
```

Update the header comment to document the new function with usage examples.

CRITICAL: The slugification must handle:
- Phase names with parentheses or special chars: "Build Skill Efficiency (QUICK)" -> "build-skill-efficiency-quick"
- Phase names with trailing dashes: strip them
- Multiple consecutive hyphens: collapse to single
- The `[COMPLETE]` tag some phase names have: strip it
  </action>
