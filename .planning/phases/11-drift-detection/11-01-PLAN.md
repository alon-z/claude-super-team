**Part A: Create drift-analysis-guide.md**

Create `plugins/claude-super-team/skills/drift/references/drift-analysis-guide.md`. This is the methodology reference embedded into the Explore agent's prompt. It must cover:

**Claim Extraction Methodology:**
- From SUMMARY.md: Extract concrete claims about files created/modified (especially from `key_files` YAML frontmatter and "Files" / "What Was Built" sections), patterns implemented, libraries used, and architectural decisions made.
- From CONTEXT.md: Extract locked decisions (technology choices, architectural patterns). These are the "contract" -- if a decision says "use library X", verify X is actually used.
- From PLAN.md: Extract file paths from `<files>` tags in tasks. These are the planned targets -- verify they exist and serve the described purpose.
- From VERIFICATION.md: If status is "passed", treat the must_haves as verified at execution time. Drift means something changed AFTER verification.
- Skip phases with no SUMMARY.md files -- they haven't been executed yet. Flag them as "unverifiable" in the report.

**Claim Verification Protocol:**
- File existence: Check if claimed files exist at their stated paths. If missing, check for renamed/moved versions via Grep on unique identifiers (function names, class names, exports).
- Pattern verification: For claims about patterns (e.g., "uses middleware pattern"), search the relevant files for the pattern. Surface-level is sufficient -- don't audit every line.
- Decision verification: For CONTEXT.md locked decisions, verify the chosen approach is present and no rejected alternative was used instead.
- Depth calibration: Focus on observable, verifiable claims. Skip vague claims like "improved error handling" -- flag them as unverifiable.

**Categorization Rules:**
- **Confirmed drift**: File is missing with no replacement found, OR a locked decision was contradicted (different library/pattern used), OR a planned file serves a clearly different purpose than described.
- **Potential drift**: File exists but content differs significantly from description, OR a pattern is partially implemented, OR renamed/moved but purpose seems preserved.
- **Aligned**: File exists at stated path with described purpose, locked decisions are reflected in code, patterns are present.
- **Unverifiable**: Claim is too vague to check, or references internal behavior that can't be observed from file inspection.

**Part B: Create SKILL.md**

Create `plugins/claude-super-team/skills/drift/SKILL.md` with this structure:

**Frontmatter:**
```yaml
---
name: drift
description: "Compare actual codebase state against planning artifacts to surface divergence. Reads SUMMARY.md, CONTEXT.md, PLAN.md files and spawns Explore agents to verify claims. Produces a categorized DRIFT-REPORT.md with confirmed drift, potential drift, and aligned findings."
argument-hint: "[phase number | --all]"
allowed-tools: Read, Write, Edit, Glob, Grep, Task, AskUserQuestion, Bash(test *), Bash(ls *), Bash(grep *), Bash(bash *gather-data.sh)
model: sonnet
---
```

Use `model: sonnet` because the orchestrator's job is coordination (reading artifacts, parsing claims, spawning agents, assembling report) -- not deep reasoning. The Explore agents spawned via Task will use opus for the actual analysis.

**Step 0: Load Context** -- Follow the established pattern. Run gather-data.sh, parse sections.

**Process:**

Phase 1: Validate Environment. Require PROJECT.md and ROADMAP.md. If no phase directories or no SUMMARY.md files anywhere, exit with guidance: "No executed phases found. Run /execute-phase first."

Phase 2: Parse Arguments. Accept:
- A phase number -> analyze that single phase
- `--all` or no argument -> analyze all completed phases (those with at least one SUMMARY.md)
- If a specific phase is given but has no SUMMARY.md, exit: "Phase {N} has not been executed. No claims to verify."

Phase 3: Extract Claims. For each phase being analyzed:
1. Read all SUMMARY.md files in the phase directory. Extract file paths, patterns, decisions from "What Was Built", "Files", and YAML frontmatter `key_files`.
2. Read CONTEXT.md if it exists. Extract locked decisions from the "Implementation Decisions" section.
3. Read PLAN.md files. Extract file paths from `<files>` tags.
4. Deduplicate claims. Group by type: file-existence claims, pattern claims, decision claims.
5. Count total claims. If more than ~40 claims for a single analysis run, warn the user that this may consume significant context and ask whether to proceed or narrow scope.

Phase 4: Spawn Explore Agent(s). Read `${CLAUDE_SKILL_DIR}/references/drift-analysis-guide.md` to get the analysis methodology.

For each phase being analyzed, spawn one Task with:
- `subagent_type: "general-purpose"`
- `model: "opus"` (deep analysis requires strong reasoning)
- Prompt includes: the drift-analysis-guide content, the extracted claims for that phase, the codebase docs (from gather-data.sh CODEBASE_DOCS section if available), and instructions to write findings as structured markdown

The agent prompt must embed ALL claims inline (no @ file references -- they don't work across Task boundaries). The agent uses Read, Glob, Grep, and Bash to inspect the actual codebase and verify each claim.

If analyzing multiple phases, spawn agents in parallel (one per phase) -- the same pattern used by execute-phase for wave parallelism.

Phase 5: Assemble Report. Read `${CLAUDE_SKILL_DIR}/assets/drift-report-template.md`. Collect results from all agents. Populate the template:
- Fill summary counts
- Group findings by phase, then by category
- Write to `.planning/DRIFT-REPORT.md`

Phase 6: Present Results. Print a summary inline:

```
Drift analysis complete.

| Category | Count |
|----------|-------|
| Confirmed Drift | {N} |
| Potential Drift | {N} |
| Aligned | {N} |
| Unverifiable | {N} |

{If confirmed drift > 0:}
Top confirmed drift items:
- {top 3 items with file and claim}

Full report: .planning/DRIFT-REPORT.md

{If confirmed drift > 0:}
To address drift:
  /phase-feedback {N} -- fix a specific phase's drift
  /quick-plan -- create a drift-fix phase
```

- Planning artifacts read and claims extracted
- Explore agent(s) spawned with full context
- Each claim categorized as confirmed drift, potential drift, aligned, or unverifiable
- DRIFT-REPORT.md created with structured findings
- User sees summary with actionable next steps
